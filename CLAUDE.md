# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A self-hosted manga/media organizer. `be/` is a FastAPI + SQLModel (SQLite) backend; `fe/` is a SvelteKit SPA (Svelte 5 runes, carbon-components-svelte). Production is a single Docker image (nginx + backend under supervisord, see `Dockerfile` and `compose.yaml`). The frontend is also deployed standalone to Vercel, where **mock mode** (`?mock=1`) is the demo/preview — it must work without any backend.

## Development model: CI is the quality gate

Development happens in cloud environments; correctness is enforced by automation, not manual testing. The rules:

1. **Backend: every feature ships with API-level tests.** `.github/workflows/be-ci.yml` runs `ruff check` and `pytest --cov=. --cov-fail-under=90` — the build fails below 90% coverage. New endpoints need tests for the success path and the error paths (404/400) in `be/tests/test_api_<router>.py`.
2. **Frontend: mock data must cover every backend feature the UI uses.** Any backend endpoint consumed by the frontend must have an MSW handler in `fe/src/lib/mock.ts`, with faker-generated data matching the real response schema exactly (same fields, same types as `fe/src/lib/client/types.gen.ts`). When you add or change a backend endpoint and its frontend usage, update `mock.ts` in the same change — otherwise the Vercel preview silently breaks.
3. **Frontend has no CI workflow yet.** Quality relies on husky pre-commit hooks (prettier + eslint on staged files) and on you running `npm run check` and `npx vitest run` before pushing.

## Commands

Backend (run from `be/`):

```bash
uv sync                                   # install deps
uv run pytest -q                          # full test suite
uv run pytest tests/test_api_comics.py::TestGetAll::test_empty   # single test
uv run pytest --cov=. --cov-report=term-missing --cov-fail-under=90  # what CI runs
uv run ruff check                         # lint (also a CI gate)
uv run uvicorn main:app --port 8001       # dev server (docs at /api/docs)
```

Frontend (run from `fe/`):

```bash
npm install
npm run dev          # vite dev server on port 8002
npm run check        # svelte-check (type errors)
npx vitest run       # unit tests
npm run lint         # prettier --check + eslint
npm run format       # prettier --write
npm run api          # regenerate the OpenAPI client (see below)
```

Cloud-environment gotcha: `fe/package-lock.json` resolves packages against `registry.npmmirror.com`. If that mirror is blocked, temporarily rewrite the registry (`sed -i 's|registry.npmmirror.com|registry.npmjs.org|g' package-lock.json`), install, then `git checkout package-lock.json` — never commit the rewritten lockfile.

## Generated API client — never hand-edit

`fe/src/lib/client/*.gen.ts` is generated by `@hey-api/openapi-ts` from the running backend's OpenAPI schema. After any backend API change:

1. Start the backend on port 8001.
2. `cd fe && npm run api`
3. Revert the generated `baseUrl` change: `git checkout src/lib/client/client.gen.ts` (it must stay `baseUrl: ''`; the real URL is set at runtime by the connect flow).

`npm run check` currently reports a few pre-existing errors inside `client.gen.ts` (hey-api package version mismatch) — don't chase them as part of unrelated work.

## Backend architecture

- `main.py` wires class-based routers (`fastapi-utils` CBV) from `api/`. `comics`, `videos`, `system` routers get `require_auth` at include time; `comicpage` and `images` declare auth per-endpoint because browser `<img>` loads can't send headers — media endpoints accept `?token=` via `require_media_auth` (`core/auth.py`).
- Auth is a single `ADMIN_PASSWORD` env var; the token is a static HMAC of it. No users, no expiry. When `ADMIN_PASSWORD` is unset, auth is disabled (this is how tests run).
- `comicfile.py` abstracts comic sources (zip / rar / directory) behind one interface, with a small TTL cache around `create_open()`.
- Persistence is split: `ComicEntity` and `VideoEntity` are SQLModel tables (SQLite via `db.py`); **`ImageEntity` lives only in an in-memory dict** in `api/images.py`, rebuilt by `bootstrap()` on scan — image-folder state does not survive restarts.
- Covers are pre-generated JPEGs named `{id}_0.jpg` written to nginx-served directories (`global_data.Config.nginx_*_path`); the frontend loads them from the static server, not the API.
- `core/exceptions.py` maps request-validation errors to **400, not 422** — write test assertions accordingly.
- Background tasks live in `tasks/` and start from `core/lifespan.py`.

## Backend test conventions

Fixtures in `be/tests/conftest.py`: `client` (TestClient with in-memory SQLite and patched nginx paths), `session`, helpers `insert_comic` / `insert_video`, and `MockComicfile` (fake archive; patch `comicfile.create_open` to avoid touching the filesystem). Page numbers are 1-based in the API and 0-based inside `Comicfile.read()`.

## Frontend architecture

- State modules use Svelte 5 runes in `*.svelte.ts` files (`config.svelte.ts`, `auth.svelte.ts`, `connection.svelte.ts`, `viewerState.svelte.ts`).
- The app is connection-gated: `ConnectDialog` either connects to a real backend URL (persisted in localStorage) or starts MSW mock mode via URL params `?mock=1` (`&auth=1` simulates login; mock password is `mock`).
- `routes/[mediaType=mediaType]/` serves comic/video/image lists from one route tree; media-type-specific behavior branches on `MediaType` (`lib/model.svelte.ts`).
- The reader (`lib/components/FileContent.svelte`) is shared by comics and image albums.
